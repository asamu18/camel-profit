# 05｜运行、构建与部署

## 前置要求

- Node.js + npm（仓库使用 `package-lock.json`，建议用 npm）
- 一个可用的 Supabase 项目（包含对应表与 RLS 策略）
- 可选：Android Studio / Xcode（做原生壳编译与真机调试）

## 环境变量配置

本地运行依赖 `.env`（见 [/.env](file:///workspace/.env)），至少需要：

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_KEY`

注意：

- 这两个变量在构建时注入前端 bundle；不要在前端写入任何服务端密钥。

## Web 本地开发

依赖与启动脚本定义于 [package.json](file:///workspace/package.json#L6-L10)：

```bash
npm install
npm run dev
```

## Web 构建与预览

```bash
npm run build
npm run preview
```

产物目录：

- `dist/`（同时也是 Capacitor 壳的 webDir，见 [capacitor.config.json](file:///workspace/capacitor.config.json#L1-L5)）

## Web 生产部署（Vercel）

仓库包含 [vercel.json](file:///workspace/vercel.json)：

- 把 `/api/supabase/:match*` rewrite 到 Supabase 项目域名
- 前端在 `PROD && !isApp` 时把 Supabase baseUrl 指向同域 `/api/supabase`（见 [supabase.js](file:///workspace/src/lib/supabase.js#L10-L13)）

建议部署检查项：

- Vercel 环境变量是否配置了 `VITE_SUPABASE_URL`、`VITE_SUPABASE_KEY`
- Supabase 的 RLS policy 是否允许当前前端用 anon key 正常读写

## 移动端（Capacitor）构建与运行

### 通用流程

1. 构建 Web 产物

```bash
npm run build
```

2. 同步到原生工程（常规做法，仓库当前未提供脚本，但可用 Capacitor CLI）

```bash
npx cap sync
```

### Android

- 工程目录：`android/`
- 主入口 Activity：[MainActivity.java](file:///workspace/android/app/src/main/java/com/camelprofit/app/MainActivity.java)

命令行构建示例：

```bash
cd android
./gradlew assembleDebug
```

### iOS

- 工程目录：`ios/`
- App 入口：[AppDelegate.swift](file:///workspace/ios/App/App/AppDelegate.swift)

常规做法：用 Xcode 打开 `ios/App/App.xcodeproj` 进行编译/签名/真机调试。

## 测试

- Web/前端：当前 [package.json](file:///workspace/package.json#L6-L10) 未配置 `test` 脚本，也未引入 Vitest/Jest
- Android：存在示例测试
  - 单元测试：`cd android && ./gradlew test`
  - 仪器测试：`cd android && ./gradlew connectedAndroidTest`（需要设备/模拟器）
